## 第1章: セールス・CRM自動化

### 成約率42%→70%への道筋

**Phase 1（Week 1-2）: 成約率42%→55%**
- 面談必須質問チェックリスト（5つの核心質問）
- 二者択一クロージングスクリプト（「お待ちしております」を禁止ワードに）
- 面談タイマー制度（40分超えたら「失敗面談」と定義）
- 比較フレーム資料（日本の私立大学880万 vs 台湾留学589万 = 291万円お得）

**Phase 2（Week 3-4）: 成約率55%→65%**
- 保護者面談3層構造（事前準備→面談中→面談後フォロー）
- 3-7-14フォロー自動化（ウタゲステップ配信）
- 先延ばし対策インセンティブ（当日決断→オンライン中国語3ヶ月無料）

**Phase 3（Week 5-8）: 成約率65%→70%+**
- リードスコアリング（100点満点、70点以上=ホットリード）
- Stripe決済自動化（Payment Link+Webhook+自動督促）
- 休眠リード掘り起こしフロー

### 面談必須質問チェックリスト

```
① 「今の進路について、一番不安に感じていることは何ですか？」（現状の痛み）
② 「その不安を放置したら、1年後どうなっていると思いますか？」（痛みの拡大）
③ 「周りの友達と比べて、焦りを感じることはありますか？」（社会的比較）
④ 「ご両親はどう思っていらっしゃいますか？」（保護者の温度感確認）
⑤ 「もし費用が全くかからないとしたら、台湾留学に行きたいですか？」（意思の分離）
```

### 保護者対策

最も重要な発見: **高校生が対象=お金を払うのは保護者**。現在の面談設計にこの構造が反映されていない。

- 面談予約時に保護者同席を必ず確認
- 保護者向け専用資料（安全性・費用対効果・就職実績・サポート体制）
- 「保護者に相談します」への対応スクリプト完備

## 第5章: テクノロジー・AI統合戦略

### 現在のアーキテクチャ評価

**強み:**
- GAS（Google Apps Script）をハブとした低コスト自動化基盤が確立
- Claude Code CLI + MCP による高度なブラウザ自動化が実現済み
- 4つの稼働中システムが安定運用されている
- 設計書18本の蓄積（設計→実装のパイプラインがある）

**弱み:**
- **データのサイロ化**: Asana / freee / Stripe / UPSIDER / クラウドサイン / ウタゲが独立動作
- **未接続API**: freee / Stripe / Chatwork / クラウドサイン / UTAGE の5つが未連携
- **GASの実行制限**: 6分タイムアウト、日次トリガー上限
- **監視体制の欠如**: エラー時に気づく仕組みがない
- **Windows環境の制約**: Node.jsパス問題、日本語エンコーディング問題

### API統合の優先順位

| 4 | **クラウドサイン API** | 中 | 中 | 契約→決済自動パイプラインの前提 |
| 5 | **UTAGE API** | 中 | 高 | ファネルKPI自動取得（APIが限定的） |

### GAS vs Node.js の使い分け基準

```
GASを使うべきケース:
  ✓ Google Workspace連携（Gmail/Drive/Sheets）
  ✓ 5分毎〜日次の定期実行
  ✓ 処理時間6分以内
  ✓ 外部Webhookの受信（doPost）
  ✓ 追加コストゼロが必須

Node.jsに移行すべきケース:
  ✗ 処理時間が6分を超える（大量データ処理）
  ✗ WebSocket/常時接続が必要（リアルタイム通知）
  ✗ SQLiteなどのローカルDB活用
  ✗ Claude Code CLIとの連携
  ✗ 複雑なエラーハンドリング・リトライ

判定: 現状の業務規模（社員4名）ではGAS中心で十分。
     Node.jsはClaude Office + SNSエージェントチームのみに限定。
```
